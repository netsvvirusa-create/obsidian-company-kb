## ГОСТ 34.602 -- Техническое задание на создание автоматизированной системы

### 1. Общие сведения

#### 1.1. Наименование системы

Автоматизированная система "Корпоративная база знаний" (obsidian-company-kb).

#### 1.2. Заказчик и разработчик

**Заказчик:** ООО "Цифратроника"
**Разработчик:** ООО "Цифратроника"

#### 1.3. Перечень документов для создания системы

- Спецификация Agent Skills (agentskills.io)
- Obsidian Flavored Markdown specification
- Obsidian Bases format specification
- JSON Canvas specification

#### 1.4. Плановые сроки

**Начало работ:** 2026 Q1
**Окончание работ:** 2026 Q2
**Выпуск версии 1.0.0:** 2026-04-05

### 2. Назначение и цели создания системы

#### 2.1. Назначение

АС предназначена для автоматизации управления корпоративными данными: контрагенты, договоры, сотрудники, контакты, переговоры, проекты, операционные события, стратегические цели.

#### 2.2. Цели создания

1. Централизация корпоративных данных в едином структурированном хранилище
2. Автоматизация генерации юридических документов (.docx)
3. Обеспечение целостности данных через валидацию и синхронизацию
4. Снижение времени на подготовку договоров и отчётов
5. Обеспечение прозрачности связей между контрагентами, людьми и договорами

### 3. Характеристика объекта автоматизации

#### 3.1. Объект автоматизации

Процессы управления корпоративными данными малого/среднего предприятия:
- Ведение реестра контрагентов
- Управление договорными отношениями
- Учёт кадров и контактных лиц
- Протоколирование переговоров
- Управление проектами
- Стратегическое планирование

#### 3.2. Условия эксплуатации

- Пользователи: руководители, менеджеры, юристы, AI-агенты
- Рабочее место: ПК с Obsidian и Python
- Режим работы: однопользовательский (одно хранилище на рабочее место)
- Объём данных: до 10 000 заметок

### 4. Требования к системе

#### 4.1. Требования к системе в целом

**Требования к структуре:** Модульная архитектура с независимыми скриптами, объединёнными через SKILL.md.

**Требования к надёжности:**
- Идемпотентность операций
- Режим предварительного просмотра (dry-run)
- Резервное копирование средствами git/файловой системы

**Требования к безопасности:**
- Отсутствие сетевого взаимодействия
- Использование yaml.safe_load
- Ограничение доступа к файлам в пределах хранилища
- См. `docs/SECURITY.md`

#### 4.2. Требования к функциям

См. раздел "Функции системы" ниже.

### 5. Состав и содержание работ

См. таблицу стадий в разделе ГОСТ 34.601.

### 6. Порядок контроля и приёмки

- Автоматическое тестирование: `make test` (покрытие >= 80%)

## Функции системы

### Перечень автоматизируемых функций

| N | Функция | Подсистема | Модуль |
|---|---------|------------|--------|
| Ф1 | Инициализация хранилища | Управление данными | `init_vault.sh` |
| Ф2 | Создание/редактирование заметок | Управление данными | AI-агент + SKILL.md |
| Ф3 | Генерация договора .docx | Документооборот | `generate_contract.py` |
| Ф4 | Генерация спецификации .docx | Документооборот | `generate_specification.py` |
| Ф5 | Грамматическая проверка | Документооборот | `grammar_check.py` |
| Ф6 | Валидация хранилища | Контроль качества | `validate_vault.py` |
| Ф7 | Аудит ссылок | Контроль качества | `audit_links.py` |
| Ф8 | Импорт из CSV | Ввод данных | `import_csv.py` |
| Ф9 | Импорт из vCard | Ввод данных | `import_vcard.py` |
| Ф10 | Формирование отчётов | Аналитика | `generate_report.py` |
| Ф11 | Массовое обновление статусов | Управление данными | `bulk_status_update.py` |
| Ф12 | Синхронизация связей | Контроль качества | `relationship_sync.py` |
| Ф13 | Отображение дашбордов | Представление | 9 файлов `.base` |
| Ф14 | Визуальные карты | Представление | 4 файла `.canvas` |

### Описание функций

**Ф1. Инициализация хранилища.** Создание структуры папок, копирование дашбордов и канвасов, генерация индексных заметок. Идемпотентная операция.

**Ф2. Управление заметками.** AI-агент определяет тип заметки по описанию пользователя, генерирует файл по шаблону, обеспечивает взаимные ссылки.

**Ф3-Ф4. Генерация документов.** Сборка словаря переменных из frontmatter трёх заметок (договор, компания, контрагент), подстановка в .docx шаблон с обработкой Word runs.

**Ф5. Грамматическая проверка.** Морфологический анализ юридических фраз: родительный падеж после "в лице" и "на основании", согласование по роду.

**Ф6-Ф7. Контроль качества данных.** Проверка обязательных полей, таксономии тегов, целостности ссылок, дубликатов, просроченных договоров.

**Ф8-Ф9. Импорт данных.** Автоматическое сопоставление колонок CSV с полями шаблонов, парсинг vCard, проверка дубликатов, обновление карточек контрагентов.

**Ф10. Отчёты.** Три типа: истекающие договоры, история контрагента, активность сотрудника.

**Ф11. Массовые операции.** Обновление поля "статус" с фильтрацией, записью в историю и автоматическим архивированием.

**Ф12. Синхронизация связей.** Проверка и восстановление двусторонних ссылок между карточками людей.

## Информационное обеспечение

### 1. Состав информации

#### 1.1. Входная информация

| Источник | Формат | Описание |
|----------|--------|----------|
| .md файлы | Markdown + YAML frontmatter | Заметки хранилища (11 типов) |
| .docx шаблоны | Office Open XML | Шаблоны документов с переменными |
| CSV файлы | UTF-8, разделитель запятая | Данные контрагентов/контактов/сотрудников |
| vCard файлы | vCard 3.0/4.0 | Контактные данные |

#### 1.2. Внутренняя информация

| Объект | Формат | Описание |
|--------|--------|----------|
| YAML frontmatter | YAML | Структурированные метаданные заметок |
| Теги | Иерархические строки | `тип/`, `статус/`, `приоритет/`, `направление/`, `связь/` |
| Wikilinks | `[[имя]]` | Связи между заметками |
| Переменные .docx | `{{VAR\|ID\|default}}` | Идентификаторы для подстановки |

#### 1.3. Выходная информация

| Результат | Формат | Описание |
|-----------|--------|----------|
| Договоры | .docx | Заполненные Word-документы |
| Спецификации | .docx | Спецификации с таблицами услуг |
| Отчёты | Markdown | Аналитические отчёты |
| Результаты валидации | JSON | Ошибки и предупреждения |
| Результаты синхронизации | JSON | Найденные/исправленные несоответствия |

### 2. Структура базы данных

База данных реализована как файловая система Obsidian vault с 15 папками. Каждый файл -- запись с YAML-метаданными. Индексация и поиск выполняются Obsidian (для UI) и Python-скриптами (для автоматизации).
