## ГОСТ Р 57580.1-2017 -- Защита информации финансовых организаций

### Применимость

Система обрабатывает финансовую информацию (суммы договоров, банковские реквизиты, ИНН/КПП). Требования ГОСТ Р 57580.1-2017 применяются в объёме, соответствующем уровню защиты информации "Стандартный" (УЗ-2).

### Процессы защиты информации

| Процесс | Реализация | Оценка |
|---------|-----------|--------|
| ЗИ.1 Управление доступом | Файловая система ОС, однопользовательский режим | Соответствует |
| ЗИ.2 Защита вычислительных сетей | Отсутствие сетевого взаимодействия в скриптах | Соответствует |
| ЗИ.3 Контроль целостности | `validate_vault.py`, `audit_links.py`, `relationship_sync.py` | Соответствует |
| ЗИ.4 Защита от вредоносного кода | `yaml.safe_load`, нет `eval`/`exec`, нет `shell=True` | Соответствует |
| ЗИ.5 Предотвращение утечек | Нет сетевого вывода, конфиденциальные данные в callouts | Соответствует |
| ЗИ.6 Управление инцидентами | Процедура реагирования (см. ниже) | Соответствует |
| ЗИ.7 Защита среды виртуализации | Не применимо (файловая система) | Н/П |
| ЗИ.8 Защита при удалённом доступе | Не применимо (локальная работа) | Н/П |

## Матрица классификации данных

| Категория | Уровень | Примеры данных | Размещение | Меры защиты |
|-----------|---------|----------------|------------|-------------|
| **Открытые** | УЗ-0 | Названия контрагентов, названия проектов, теги | YAML frontmatter, тело заметки | Без дополнительных мер |
| **Служебные** | УЗ-1 | Суммы договоров, условия, сроки, статусы | YAML frontmatter | Контроль доступа на уровне ОС |
| **Конфиденциальные** | УЗ-2 | ИНН, КПП, ОГРН, банковские реквизиты (расчётный счёт, БИК, корр. счёт) | YAML frontmatter + `> [!info]-` callout | Шифрование диска, ограниченный доступ |
| **Персональные данные** | УЗ-3 | Личные телефоны, домашние адреса, паспортные данные, СНИЛС | `> [!warning]-` сворачиваемый callout | Шифрование, минимизация, согласие субъекта |

### Правила обращения с данными по категориям

**Открытые:** Могут отображаться в дашбордах, канвасах и отчётах без ограничений.

**Служебные:** Отображаются в дашбордах для авторизованных пользователей. В отчётах -- с пометкой "Для служебного пользования".

**Конфиденциальные:** Хранятся в YAML frontmatter. В теле заметки -- в сворачиваемых callouts `> [!info]-`. Не включаются в отчёты без явного запроса.

**Персональные данные:** Хранятся исключительно в сворачиваемых callouts `> [!warning]- Конфиденциальная информация`. Не индексируются дашбордами. Обработка только с согласия субъекта ПДн.

## Матрица управления доступом

### Ролевая модель

| Роль | Описание | Права |
|------|----------|-------|
| Администратор | Владелец vault | Полный доступ |
| Менеджер | Ответственный за направление | Чтение/запись своих данных |
| Юрист | Работа с договорами | Чтение/генерация .docx |
| Наблюдатель | Ознакомительный доступ | Только чтение открытых данных |
| AI-агент | Автоматизированная обработка | По правилам SKILL.md |

### Матрица прав доступа по операциям

| Операция | Администратор | Менеджер | Юрист | Наблюдатель | AI-агент |
|----------|:---:|:---:|:---:|:---:|:---:|
| Чтение заметок (все) | + | + | + | +* | + |
| Чтение ПДн | + | +** | + | - | - |
| Создание заметок | + | + | + | - | + |
| Редактирование заметок | + | +** | +** | - | + |
| Генерация .docx | + | + | + | - | + |
| Удаление заметок | + | - | - | - | - |
| Массовое обновление статусов | + | - | - | - | + |
| Импорт данных | + | + | - | - | + |
| Валидация vault | + | + | + | + | + |
| Настройка vault (init) | + | - | - | - | + |

`*` -- только открытые данные
`**` -- только свои данные/направление

### Реализация контроля доступа

Контроль доступа реализуется на уровне файловой системы ОС:
- Права на чтение/запись папок vault
- Разделение доступа через группы ОС
- Для AI-агента -- через правила SKILL.md (раздел "Agent behavior")

## Требования к шифрованию

### Шифрование в покое (at rest)

| Компонент | Рекомендация | Реализация |
|-----------|-------------|------------|
| Vault на диске | Полнодисковое шифрование | BitLocker (Windows), FileVault (macOS), LUKS (Linux) |
| .docx документы | Хранение в зашифрованной папке | Шифрованный контейнер или зашифрованная папка |
| Резервные копии | Шифрование архивов | GPG/7-zip с AES-256 |
| USB-носители | Аппаратное или программное шифрование | BitLocker To Go, VeraCrypt |

### Шифрование при передаче (in transit)

| Канал | Рекомендация |
|-------|-------------|
| Git remote | SSH или HTTPS с TLS 1.2+ |
| Obsidian Sync | Встроенное E2EE (end-to-end encryption) |
| Файловый обмен | SFTP, SCP или зашифрованные архивы |
| Электронная почта | PGP/S-MIME для вложений с ПДн |

### Управление ключами

- Ключи шифрования хранятся отдельно от зашифрованных данных
- Смена паролей шифрования -- не реже 1 раза в год
- Резервные копии ключей -- в защищённом хранилище

## Процедуры реагирования на инциденты

### Классификация инцидентов

| Уровень | Описание | Примеры | Время реагирования |
|---------|----------|---------|-------------------|
| Критический | Утечка ПДн или финансовых данных | Несанкционированный доступ к vault | 1 час |
| Высокий | Нарушение целостности данных | Массовое повреждение заметок | 4 часа |
| Средний | Нарушение доступности | Потеря части данных, битые ссылки | 24 часа |
| Низкий | Незначительные нарушения | Опечатки в тегах, пропущенные поля | 72 часа |

### Процедура реагирования

#### Этап 1: Обнаружение

Средства обнаружения:
- `validate_vault.py` -- выявляет аномалии в данных (отсутствующие поля, невалидные теги, дубликаты)
- `audit_links.py` -- обнаруживает битые ссылки
- `relationship_sync.py` -- находит несогласованные связи
- Мониторинг файловой системы (опционально)

Рекомендация: запускать `validate_vault.py` ежедневно или при каждом существенном изменении.

#### Этап 2: Классификация

1. Определить тип затронутых данных по матрице классификации
2. Определить масштаб: количество затронутых заметок
3. Определить уровень инцидента по таблице выше
4. Назначить ответственного

#### Этап 3: Локализация

1. Идентифицировать затронутые файлы
2. При необходимости -- отключить доступ к vault
3. Сохранить копию текущего состояния (снимок файловой системы)

#### Этап 4: Устранение

1. Восстановить данные из git-репозитория:
   ```bash
   git log --oneline -20          # найти коммит до инцидента
   git diff HEAD~N -- path/to/file  # проверить изменения
   git checkout HEAD~N -- path/to/file  # восстановить файл
   ```
2. Или восстановить из резервной копии
3. Запустить валидацию:
   ```bash
   python scripts/validate_vault.py --vault /path/to/vault
   python scripts/audit_links.py --vault /path/to/vault
   python scripts/relationship_sync.py --vault /path/to/vault --dry-run
   ```

#### Этап 5: Анализ

1. Определить причину инцидента
2. Проверить журналы (git log, системные логи)
3. Оценить, были ли данные скомпрометированы

#### Этап 6: Документирование

Создать заметку типа `событие` в `07-ОПЕРАЦИИ/`:

```yaml
---
title: "Инцидент ИБ: Описание"
type: событие
дата: 2026-04-05
категория: инцидент
направление: it
приоритет: высокий
статус: завершён
tags:
  - тип/событие
  - приоритет/высокий
---
```

Содержание: описание инцидента, хронология, затронутые данные, принятые меры, выводы и рекомендации.

### Уведомления

| Событие | Кого уведомлять | Срок |
|---------|-----------------|------|
| Утечка ПДн | Роскомнадзор, субъекты ПДн | 24 часа (ФЗ-152) |
| Утечка финансовых данных | Руководство, СБ | 4 часа |
| Нарушение целостности | Администратор vault | 24 часа |

## Рекомендации по усилению безопасности

1. **Версионирование:** хранить vault в git-репозитории для отслеживания всех изменений
2. **Резервное копирование:** ежедневные автоматические бэкапы с шифрованием
3. **Аудит доступа:** периодическая проверка прав доступа к папке vault
4. **Обновление зависимостей:** регулярная проверка уязвимостей в Python-пакетах (`pip audit`)
5. **Обучение:** инструктаж пользователей по работе с конфиденциальными данными
6. **Минимизация ПДн:** хранить только необходимые персональные данные
7. **Автоматическая валидация:** включить `validate_vault.py` в CI/CD или cron
