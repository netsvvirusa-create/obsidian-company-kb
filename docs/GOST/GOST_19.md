## ГОСТ 19.201 -- Техническое задание

### 1. Введение

**Наименование программы:** obsidian-company-kb

**Краткая характеристика области применения:** Управление корпоративной базой знаний (контрагенты, договоры, сотрудники, контакты, переговоры, проекты, операции, стратегия) в среде Obsidian с автоматической генерацией .docx документов.

### 2. Основание для разработки

Разработка ведётся в рамках создания инструментария корпоративного управления ООО "Цифратроника" на основе спецификации Agent Skills (agentskills.io).

### 3. Назначение разработки

**Функциональное назначение:** Автоматизация ведения корпоративной базы знаний, включая:
- Структурированное хранение информации о контрагентах, договорах, сотрудниках
- Генерация юридических документов (.docx) из шаблонов с подстановкой данных
- Проверка грамматической корректности юридических документов
- Импорт данных из внешних источников (CSV, vCard)
- Валидация целостности данных хранилища
- Формирование отчётов

**Эксплуатационное назначение:** Программа предназначена для использования в качестве skill-расширения AI-агентов (Claude Code, Codex CLI, OpenCode) при работе с Obsidian vault.

### 4. Требования к программе

#### 4.1. Требования к функциональным характеристикам

Программа должна обеспечивать:

1. **Инициализацию хранилища** -- создание 15 папок, 9 дашбордов, 4 канвасов, MOC-заметок
2. **Управление 15 типами заметок** -- с YAML-frontmatter и иерархической системой тегов
3. **Генерацию .docx** -- договоры (32 переменные) и спецификации (вложенные переменные, динамические таблицы)
4. **Грамматический контроль** -- проверка падежей и согласования в юридических документах
5. **Импорт данных** -- из CSV, vCard и протоколов встреч с автоматическим сопоставлением колонок
6. **Валидацию данных** -- проверка схемы, тегов, ссылок, дубликатов
7. **Синхронизацию связей** -- поддержание двусторонних ссылок между карточками людей
8. **Формирование отчётов** -- истекающие договоры, история контрагента, активность сотрудника
9. **Быстрый захват** -- создание заметок из кратких описаний (`quick_capture.py`)
10. **Ежедневные операции** -- создание дневных заметок, утренние брифинги, контроль просроченных (`daily_operations.py`)
11. **Периодический синтез** -- генерация еженедельных и ежемесячных ретроспектив (`periodic_synthesis.py`)
12. **Синхронизацию MOC** -- обновление индексных заметок при изменении содержимого хранилища (`sync_moc.py`)
13. **Генерацию canvas** -- создание визуальных карт из данных хранилища (`generate_canvas.py`)
14. **Управление архивом** -- сканирование кандидатов на архивирование и перемещение завершённых элементов (`archive_manager.py`)

#### 4.2. Требования к надёжности

- Идемпотентность операций инициализации (повторный запуск не разрушает данные)
- Режим `--dry-run` для предварительного просмотра изменений
- Валидация входных данных перед записью
- Обработка ошибок с информативными сообщениями

#### 4.3. Требования к составу и параметрам технических средств

- ОС: Windows, macOS, Linux
- Python 3.10+
- Оперативная память: не менее 256 МБ
- Дисковое пространство: не менее 50 МБ (без учёта данных хранилища)

#### 4.4. Требования к информационной и программной совместимости

- Формат данных: Obsidian Flavored Markdown с YAML frontmatter
- Формат дашбордов: Obsidian Bases YAML (.base)
- Формат визуальных карт: JSON Canvas (.canvas)
- Формат документов: Office Open XML (.docx)
- Зависимости: pyyaml, python-docx, pymorphy3, pymorphy3-dicts-ru, vobject

### 5. Требования к программной документации

Документация включает: описание программы, руководство оператора, руководство программиста, руководство системного программиста.

### 6. Стадии и этапы разработки

| Стадия | Этап | Результат |

### 7. Порядок контроля и приёмки

- Прохождение статического анализа (ruff, mypy)
- Проверка безопасности (ruff --select S)

## ГОСТ 19.301 -- Эскизный проект

### 1. Общие положения

Программа представляет собой набор Python-скриптов и bash-скрипта, объединённых определением skill (SKILL.md) для AI-агентов. Данные хранятся в файлах Obsidian vault в формате Markdown с YAML-метаданными.

### 2. Архитектурные решения

Программа построена на трёхуровневой архитектуре:

1. **Уровень представления** -- Obsidian (заметки, дашборды, канвасы)
2. **Уровень логики** -- Python-скрипты (19 модулей)
3. **Уровень данных** -- YAML frontmatter в .md файлах

Подробная диаграмма архитектуры -- в `docs/ARCHITECTURE.md`.

### 3. Основные компоненты

- **SKILL.md** -- точка входа для AI-агента, определяет workflow
- **references/** -- справочные документы (шаблоны, теги, переменные)
- **scripts/** -- исполняемая логика (18 Python + 1 bash)
- **assets/** -- статические ресурсы (шаблоны .docx, файлы инициализации)

### 4. Потоки данных

```
Ввод (CSV/vCard/ручной) → .md заметки → Python-скрипты → .docx / Отчёты / Валидация
```

## ГОСТ 19.401 -- Технический проект (описание проектных решений)

### 1. Структура данных

#### 1.1. Типизация заметок

Каждая заметка содержит поле `type` в YAML frontmatter. Определено 15 типов: `контрагент`, `договор`, `проект`, `сотрудник`, `контакт`, `переговоры`, `событие`, `дневная_запись`, `цель`, `идея`, `наша_компания`, `платёж`, `счёт`, `бюджет`, `ретроспектива`.

#### 1.2. Система тегов

Иерархическая таксономия с 5 префиксами: `тип/`, `статус/`, `приоритет/`, `направление/`, `связь/`. Теги хранятся в YAML-массиве `tags` без символа `#`.

#### 1.3. Ссылочная модель

Связи между сущностями реализованы через `[[wikilinks]]` в YAML-свойствах. Двусторонние связи обеспечиваются автоматической синхронизацией.

### 2. Алгоритм генерации .docx

1. Чтение frontmatter заметки договора (`yaml.safe_load`)
2. Разрешение wikilinks на контрагента и компанию (поиск по `stem`)
3. Сборка словаря переменных (32 для договора, +5 для спецификации)
4. Рекурсивное разрешение вложенных переменных (для спецификаций)
5. Подстановка в .docx с учётом структуры Word runs
6. Обработка таблиц, колонтитулов
7. Проверка незаменённых переменных
8. Опциональная грамматическая проверка

### 3. Алгоритм валидации

1. Сканирование всех .md файлов в хранилище
2. Парсинг frontmatter каждого файла
3. Проверка обязательных полей (общих и по типу)
4. Валидация тегов по таксономии
5. Проверка существования целей wikilinks
6. Обнаружение дубликатов (ИНН, номера договоров)
7. Проверка просроченных активных договоров
8. Проверка двусторонних связей

## ГОСТ 19.402 -- Описание программы

### 1. Общие сведения

**Наименование:** obsidian-company-kb
**Версия:** 1.0.0
**Язык:** Python 3.10+, bash
**Лицензия:** MIT

### 2. Функциональное назначение

Программа предназначена для управления корпоративной базой знаний в формате Obsidian vault. Обеспечивает структурированное хранение данных о контрагентах, договорах, сотрудниках, контактах, переговорах, проектах, операционных событиях и стратегических целях.

### 3. Описание логической структуры

Программа состоит из 19 исполняемых модулей. Подробное описание каждого модуля -- в `docs/MODULES.md`.

### 4. Используемые технические средства

Python 3.10+, Obsidian, AI-агент (Claude Code / Codex CLI / OpenCode).

### 5. Вызов и загрузка

Скрипты вызываются из командной строки:
```bash
python scripts/{имя_скрипта}.py --vault /путь/к/хранилищу [параметры]
bash scripts/init_vault.sh /путь/к/хранилищу
```

### 6. Входные данные

- .md файлы хранилища с YAML frontmatter
- .docx шаблоны с переменными `{{VAR|ID|default}}`
- CSV файлы с данными контрагентов/контактов/сотрудников
- vCard файлы с контактными данными

### 7. Выходные данные

- .docx документы (договоры, спецификации)
- .md заметки (при импорте)
- JSON-отчёты (валидация, синхронизация)
- Markdown-отчёты (истекающие договоры, история, активность)

## ГОСТ 19.504 -- Руководство программиста

### 1. Назначение и условия применения

Программа предназначена для расширения и модификации разработчиками. Для разработки требуется Python 3.10+ и инструменты разработки (ruff, pytest, mypy).

### 2. Характеристика программы

- **Язык:** Python 3.10+ (type hints, pathlib, f-strings)
- **Стиль кода:** Google docstrings, ruff formatter
- **Архитектура:** модульная (каждый скрипт -- самостоятельный CLI)
- **Тестирование:** pytest с маркерами unit/integration/functional

### 3. Обращение к программе

Все модули имеют двойной интерфейс:
- **CLI** -- `argparse`-based command line interface
- **API** -- импортируемые функции для программного вызова

Пример программного вызова:
```python
from scripts.generate_contract import generate_contract
from pathlib import Path

result = generate_contract(
    vault=Path("/path/to/vault"),
    contract_name="Договор №001",
    dry_run=True,
)
```

### 4. Входные и выходные данные

Функции принимают типизированные аргументы (`Path`, `str`, `bool`) и возвращают `dict[str, Any]` с результатами операции. Ключ `"success"` указывает на успешность выполнения.

### 5. Сообщения

Логирование через `logging` модуль. Уровни: DEBUG (при `-v`), INFO (по умолчанию), WARNING, ERROR.

### 6. Расширение

Для добавления нового типа заметки:
1. Добавить шаблон в `references/TEMPLATES.md`
2. Добавить обязательные поля в `REQUIRED_FIELDS_BY_TYPE` в `validate_vault.py`
3. Добавить тег в `references/TAGS.md`
4. Обновить таблицу в `SKILL.md`

## ГОСТ 19.505 -- Руководство системного программиста

### 1. Общие сведения о программе

obsidian-company-kb -- прикладная программа (agent skill), не требующая системного программирования для установки. Работает в пользовательском пространстве без привилегий.

### 2. Структура программы на носителях

```
obsidian-company-kb/
  SKILL.md              # Определение skill
  Makefile              # Цели сборки/тестирования
  pyproject.toml        # Конфигурация проекта
  requirements.txt      # Зависимости Python
  scripts/              # Исполняемые модули
  references/           # Справочные документы
  assets/               # Статические ресурсы
  tests/                # Тесты
  docs/                 # Документация
```

### 3. Настройка программы

Программа не требует конфигурационных файлов. Все параметры передаются через аргументы командной строки.

### 4. Проверка программы

```bash
make all    # lint + test + coverage + audit
```

Или поэтапно:
```bash
make lint       # ruff check + format
make test       # pytest
make coverage   # pytest --cov (fail_under=80)
make audit      # ruff S + vulture + mypy
```

### 5. Дополнительные возможности

- **Очистка кэша:** `make clean`
- **Тесты по уровням:** `make test-unit`, `make test-integration`, `make test-functional`
- **HTML-отчёт покрытия:** `docs/coverage_html/` (генерируется `make coverage`)
